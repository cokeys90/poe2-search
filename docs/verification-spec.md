# 검증 명세

기능([feature-spec.md](feature-spec.md))마다 **무엇을 어떻게 검증하는가**. 결과는
[verification-report.md](verification-report.md)에 자동으로 남는다.

핵심 원칙 — **검증하지 못한 것은 검증했다고 말하지 않는다.**
자동 검증이 없는 기능은 아래 "수동 검증"에 절차를 적어 두고, 보고서에도 "자동으로 검증하지 못하는 것"으로 남긴다.

```bash
node scripts/verify-all.mjs          # 오프라인 검증 전부 (~30초)
node scripts/verify-all.mjs --net    # + poe2db·거래소 API 대조 (~1분)
```

## 1. 자동 검증

| ID | 검증 | 덮는 기능 | 방법 | 판정 |
|---|---|---|---|---|
| V-A | 골든 회귀 | F-02 F-03 | 대표 케이스의 생성 검색어를 전수 덤프해 베이스라인과 **바이트 비교** (옵션별 필수/선택 혼합 케이스 포함 — 전역 mode를 바꿔도 `req`가 그룹을 정해 결과가 같은지 본다) | 한 글자라도 다르면 실패 |
| V-B | 범위 정규식 경계값 | F-03 | `rangeRegex(lo,hi)`를 실제 정규식으로 되돌려 `lo-2 … hi+3` 전수 대입 | 범위 안팎 판정이 하나라도 틀리면 실패 |
| V-C | 검색조각 유효성 | F-01 F-03 | 조각을 **같이 놓이는 풀 전체**(경로석 1 + 서판 8종)에 돌린다 | 아래 4가지 전부 |
| V-D | 복합 모드 실제 옵션 줄 | F-01 | poe2db의 여러 줄 모드에서 실제 표시되는 줄을 가려낸다 | 예외(마지막 줄이 독립 옵션)를 경고 |
| V-E | 게임 용어 ↔ poe2db | F-12 | 화면의 게임 용어(접두어·타락·등급·서판·경로석)를 poe2db 원문과 대조 | 다른 낱말이면 실패 (복수형·격변화는 통과) |
| V-F | 기본 타입명 ↔ 거래소 | F-06 F-12 | 11개 거래소 API의 아이템 목록에 우리 타입명 9개가 있는가 | 한 글자라도 다르면 실패 |
| V-G | 거래소 왕복 | F-02 F-06 F-07 | 조건 → URL → 되읽기 → 원래 조건과 비교 (네트워크 없음. 필수→`and`·선택→`count`·제외→`not` 왕복 포함) + 워커 허용 목록이 앱이 부르는 거래소 11곳을 덮는가 | 조건이 하나라도 변하면 실패 / 워커가 못 덮으면 실패 |
| V-H | 250자 예산 | F-03 F-12 | 실전 조합의 검색어 길이를 10개 언어에서 실측 | 250자 초과면 실패 |
| V-I | 빌드 | F-12 F-13 | `npm run build` — 언어별 HTML 생성 + 경로 정합 검사 | 빌드 실패 / 경로 불일치면 실패 |

### V-C 검색조각 유효성 — 무엇을 보는가 (가장 중요)

조각이 "그 옵션만, 언제나" 잡는지를 본다. 조각 하나가 틀리면 **그 옵션의 매물이 통째로 사라진다.**

1. **자기 자신을 잡는가** — 굴려질 수 있는 **모든 값**에서. 특정 값에서만 놓쳐도 그 매물은 안 보인다
2. **게임 표시 양쪽에서 잡는가** — 어드밴스드 모드 설명 켬(`28(27-33)%`) / 끔(`28%`)
3. **수치를 넣은 완성된 검색어까지 잡는가** — 조각 위치가 틀리면 여기서 깨진다
4. **같은 풀의 다른 옵션을 몇 개나 잡는가** — 데이터의 `overlap`과 일치하는가

이 검증기가 없던 시절 데이터엔 **자기 옵션조차 못 잡는 조각이 5개** 있었다.

### 왜 왕복(V-G)인가

내보내기만 보면 "거래소가 알아들었는지"를 알 수 없다. 우리가 만든 JSON을 **우리 파서로 되읽어**
원래 상태와 맞춰 보면 매핑의 앞뒤가 어긋난 곳이 드러난다. (실제로 `query.type` 언어 버그를 이렇게 잡았다)

## 2. 수동 검증 — 사람이 봐야 하는 것

자동 검증은 **"우리 규칙대로 만들어졌다"까지만** 보증한다. 아래는 브라우저·게임에서 직접 봐야 한다.

👉 **직접 본 결과는 [manual-checks.md](manual-checks.md)에 커밋·날짜와 함께 남긴다.**
안 그러면 다음에 "했나 안 했나"를 모른다. 통과와 미검증을 섞지 않는다.

### M-1 인게임 검색 🔴 최종 판정자

생성한 검색어를 게임 검색창에 붙여넣어 **의도한 아이템만** 나오는지 본다.
자동 검증이 전부 통과해도 여기서 깨질 수 있다 — 게임의 검색 동작에 대한 우리 가정이 틀렸을 수 있으니까.

인게임으로 확인해 둔 동작(CLAUDE.md §3-3-1)이 이 검증의 전제다:
`.`은 줄바꿈을 안 넘는다 / `^` `\s` `\b` 동작 / 조각 끝 공백은 게임이 잘라낸다 / 모드는 한 줄로 표시된다

**게임이 업데이트되면 이 전제부터 다시 확인한다.**

### M-2 즐겨찾기 (F-08)

그룹 만들기 → 검색 저장 → 다른 그룹으로 드래그 이동 → 불러오기 → 창을 띄운 채 옵션 선택(non-modal) →
새로고침 후에도 남아 있는가

### M-3 핀 · 옵션 개인화 (F-09 F-10)

핀 켠 값이 탭을 바꿔도 유지되는가 / 옵션을 드래그해 정렬하고 숨긴 뒤 새로고침해도 그대로인가 /
설정의 "초기화"가 되돌리는가

### M-3b 필수/선택/제외 드래그 (F-02)

결과 바에서 칩을 **드래그**해 필수·선택·제외 존 사이로 옮기는 건 **네이티브 HTML5 드래그**라
자동화가 못 잡는다(합성 마우스 이벤트로는 `dragstart`/`drop`이 안 뜬다). 브라우저에서 직접 본다:
칩의 `⠿` 손잡이를 잡고 존으로 옮기면 그룹이 바뀌고 검색어·거래소 매핑이 따라 바뀌는가.
(별 토글로 선택↔필수 전환하는 로직 자체는 V-A 혼합 케이스가 덮는다)

### M-4 localStorage 이관 (F-14)

**옛 사용자의 데이터가 사라지지 않는지.** `scripts/make-legacy-fixture.mjs`로 옛 스키마를 심고
새 앱을 열어 즐겨찾기·핀·옵션순서가 그대로 보이는지 확인한다.

### M-5 다국어 화면 (F-12) — 🔴 자동 검증이 못 잡는 구멍이 여기 있다

언어를 바꿔 **옵션 원문과 화면 문구가 둘 다** 그 언어인가 / 키릴·태국어·한자가 안 깨지는가(폰트) /
언어를 바꿔도 고른 옵션이 유지되는가

⚠️ **"옵션 원문과 화면 문구가 둘 다"를 꼭 본다.** 로케일(게임 데이터)과 i18n(화면 문구)은 다른
경로로 실린다 → 한쪽만 실패하면 **옵션은 한국어인데 버튼은 영어인 반쪽 화면**이 된다.
실제로 겪었다: 동적 import에 `with { type: "json" }`을 붙였더니 **Vite dev에서만** 로케일 로딩이
깨졌다(빌드본은 JSON을 번들에 넣으므로 멀쩡). 자동 검증은 전부 통과했고, 브라우저에서 언어를
바꿔 봐야만 드러났다.

지금은 실패가 **시끄럽다** — 로케일을 먼저 받아 두고, 실패하면 아무것도 안 바꾸고
`console.error`를 남긴 뒤 이전 언어를 유지한다. 그래도 **반쪽 화면인지 눈으로 볼 것.**

### M-6 배포된 경로 (F-13)

`/en/` `/th/` 등에 직접 접속해 그 언어로 뜨는가 / 페이지 소스의 `<title>`·canonical·hreflang이 맞는가

탭별 랜딩(30개 URL)이 생긴 뒤로 함께 볼 것:

- `/waystone/` `/en/tablet/` 등이 **그 탭으로 열리는가** (주소 ≠ 화면이면 새로고침할 때마다 튕긴다)
- 탭을 고르면 주소가 따라오는가 / **허브(`/` `/en/`)는 주소가 그대로 남는가**
  (허브가 첫 화면에서 `/tablet/`으로 바뀌면 허브 URL이 색인에서 죽는다)
- hreflang이 **같은 페이지끼리** 짝지어졌는가 (`/waystone/` → `/en/waystone/`)
- ⚠️ 배포 뒤 `/en/waystone/`이 **새 앱으로 뜨는가** — `firebase.json`의 헤더는 요청 경로로 매칭하고
  `*`는 `/`를 넘지 않는다. 2단계 경로(`/*/*/`) 규칙이 빠지면 20개 URL이 1시간 동안 옛 앱을 띄운다

### M-7 북마클릿 (F-07)

실제 거래소에서 검색 → 북마클릿 클릭 → 앱이 그 조건으로 열리는가.
⚠️ 북마클릿은 **이미 배포돼 사용자 즐겨찾기에 저장돼 있다** → 하위호환을 깨면 안 된다.

## 3. 절차 — 언제 무엇을 돌리는가

| 무엇을 고쳤나 | 돌린다 |
|---|---|
| `regex.js` · `pattern.js` · 옵션 데이터 | `verify-all.mjs` (골든 diff를 눈으로 확인) + M-1 |
| 조각·로케일 | `verify-all.mjs --net` + M-1 |
| 거래소 (`trade.js`) | `verify-all.mjs --net` + M-7 |
| 화면 (컴포넌트·i18n) | `npm run build` + 해당 M-* |
| 게임 업데이트 | **전부** — CLAUDE.md §6-2의 순서대로 재생성 후 `verify-all.mjs --net` + M-1 |
| 언어 추가 | CLAUDE.md §6-2 체크리스트 + `verify-all.mjs --net` + M-5 M-6 |

의도한 변경으로 골든이 달라졌다면 베이스라인을 다시 뜨고 **diff를 커밋에 함께 올린다**:

```bash
node scripts/golden.mjs > tests/golden.baseline.txt
```

## 4. 아직 자동화하지 못한 것 (알고 있는 구멍)

정직하게 남긴다. 이걸 "통과"로 세지 않는다.

- **F-08 F-09 F-10 F-14 F-15** — 브라우저 상호작용·localStorage. 자동화하려면 e2e(Playwright)가 필요하다
- **F-07 북마클릿** — 실제 거래소 페이지가 있어야 한다. `queryToState`까지는 V-G가 덮지만,
  거래소 페이지에서 `window.tradeOpts.state`를 읽는 부분은 못 덮는다
- **M-1 인게임** — 자동화 불가. 이 앱의 정답지는 게임 안에만 있다
